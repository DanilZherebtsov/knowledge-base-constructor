# Механика: spec-lifecycle

**Что делает (1 строка):** задаёт **единицу работы** для проектов с кодом — спека/спринт в `specs/`, со статусом во frontmatter, с закрытием через извлечение ADR и синтез знаний по всем типам вики. **Исполнение** спеки (цикл написания кода) несёт механика [`software-engineering`](../software-engineering/_about.md): spec-lifecycle отвечает за «что делаем и как отслеживаем», software-engineering — за «как код пишется и проверяется». У saas подключены обе.

## Слоты целевого проекта, которые трогает

- **S6** (доменный lifecycle-файл) — *заполняет.* Это и есть содержимое слота S6 для классов с кодом: указатель `methodology/spec-lifecycle.md` в дереве «Архитектура» и в разделе «Wiki: типы и операции».
- **S4** (дополнительные рабочие слои для классов с кодом: `specs/` + `src/` + `data/`) — *требует* заведённой папки `specs/`. Эта механика — потребитель S4; саму папку объявляет слот S4 в дереве.
- **S2** (доменный тип вики) — *опирается* на наличие `architecture/` и `principles/` в `wiki/` (синтез при закрытии перебирает все 5 типов, включая их).

## Файлы целевого проекта, которые трогает

- `methodology/spec-lifecycle.md` — кладётся этот drop-in (единственный новый файл механики).
- `CLAUDE.md` — две вставки-указателя (см. wiring): блок «Поток task-спек / Спринты» в раздел «Wiki: типы и операции»; правило про `specs/` в «Архитектура». (Always-on строку про цикл исполнения вставляет механика `software-engineering`, не эта.)
- `methodology/lint.md` — одна вставка в блок «Только отчёт»: проверка дрейфа статусов спек (композиционная, шаг 12). Это НЕ слот DOMAIN-LINT (он занят центральным типом класса) — отдельный буллет, как у `claim-graph`.
- Ссылается (как на соседей в `methodology/`) на: `state-rules.md`, `ingest.md`. Они должны присутствовать (наследуются из base).
- Создаёт во время работы (не при установке): `specs/<feature>-NNN-<slug>.md`, `specs/SPRINT-<NAME>.md`, ADR в `wiki/decisions/`, записи в `STATE.md`.

## От каких механик зависит

- **base-скелет** — `wiki/{decisions,architecture,discovery,synthesis,principles}/`, `methodology/{ingest,state-rules}.md`, дисциплина «человек в петле записи». Без них механика повисает.
- **S4-слой `specs/`** — должен быть объявлен в дереве «Архитектура» целевого проекта (иначе спекам некуда ложиться).
- Самостоятельных зависимостей от `roles` или `claim-graph` нет.

## Пошаговый wiring

1. **Скопировать drop-in:** `spec-lifecycle.md` → `<target>/methodology/spec-lifecycle.md`.
2. **Заполнить слот S6.** В дереве «Архитектура» (`CLAUDE.md`) в блоке `methodology/` заменить строку-плейсхолдер `<<СЛОТ S6: доменный lifecycle-файл …>>` на:
   `spec-lifecycle.md    (pending/active/accepted/rejected)`
3. **Заполнить слот S6 в разделе «Wiki».** Заменить `<<СЛОТ S6: указатель на lifecycle-файл класса>>` на указатель потока:
   «**Поток task-спек** (бэклог в STATE → файл в `specs/` со `status: active` → извлечение ADR при acceptance, спека замораживается в `completed`) — [methodology/spec-lifecycle.md](methodology/spec-lifecycle.md).»
   Рядом добавить абзац **Спринты** (веха ≥3 task-спек → `SPRINT-<NAME>.md`, `kind: sprint`; список задач живёт в спринт-спеке, не в STATE) со ссылкой на тот же файл.
4. **Объявить `specs/` (слот S4).** Убедиться, что в дереве «Архитектура» есть top-level `specs/` с пояснением: плоско, один файл на задачу `<feature>-NNN-<slug>.md`, статус во frontmatter (НЕ в подпапках), STATE.md — индекс активных. Если S4 ещё не раскрыт под код — раскрыть (`specs/` + `src/` + `data/`).
5. **Добавить правило в «Архитектура».** В список правил под деревом вставить: «**`specs/` — рабочие документы задач, не код и не вики.** Мутабельна пока `active`; после `completed`/`rejected` заморожена. Детали — [methodology/spec-lifecycle.md](methodology/spec-lifecycle.md).»
6. **Привязать к дисциплине (правило синтеза).** Правило «Синтез знаний при закрытии единицы работы» в разделе «Дисциплина» уже есть в base; уточнить хвост: «task-спека/спринт → оценка знаний ([methodology/spec-lifecycle.md](methodology/spec-lifecycle.md))». Для классов с кодом это вторая точка закрытия (первая — research-веха → синтез-проход в `ingest.md`). (Always-on указатель на сам цикл исполнения вставляет механика `software-engineering` — здесь не дублировать.)
7. **Bootstrap/STATE.** Убедиться, что `STATE.md` содержит разделы «Путь до цели», «Сейчас в работе», «Следующее», «Завершено» — спека на них опирается (бэклог и индекс активного). Это слот S5 (`state-rules.md`); правки в саму механику не требуются. **В `bootstrap.md` (слот S4): eager-создание при bootstrap — ТОЛЬКО `specs/.gitkeep`.** `src/`/`data/`/`docs/`/`scripts/` НЕ создавать пустыми, а перечислить в разделе «Что НЕ создаётся при bootstrap» («появляются по решению человека, когда нужны»).
8. **Тело STATE (state-rules).** Вставить в base `state-rules.md` содержимое `state-rules-body.md` этой механики: заместить регион слота S5 («## Структура» с 7 dev-разделами + гранулярность) и уточнить правило 2б (перенос «Сейчас в работе» → «Завершено»). Базовые правила 1, 3–7 наследуются из base.
9. **Runtime-данные.** Правило «Runtime-данные — в `data/`, не в `wiki/`» вставляет механика `software-engineering` (слой `data/` — часть владения кодом). Здесь не дублировать.
10. **Закрытие research-вехи (синтез-проход) в `ingest.md`.** saas закрывает работу в двух точках: task-спека/спринт (выше) И research-веха. Дописать в `ingest.md` секцию «## Закрытие research-вехи: синтез-проход» — drop-in `ingest-closure-section.md` этой механики (сведение знания поперёк источников при закрытии вехи discovery; симметрично закрытию спеки). Это снимает мёртвые ссылки на секцию из `spec-lifecycle.md` и Дисциплины #8.
11. **Чего НЕ добавлять.** Не дублировать `specs/active|completed|rejected/` подпапками — статус живёт во frontmatter. Не заводить параллельный список задач в STATE — он в спринт-спеке.
12. **`lint.md` → проверка дрейфа статусов спек (композиционная).** В блок «## Только отчёт (эвристическое — требует суждения человека)», отдельным буллетом (НЕ в слот DOMAIN-LINT — он занят центральным типом класса), добавить:

    ```markdown
    - **Дрейф статусов спек.** Спека `status: active`, работа по которой закрыта, тихо копится и искажает
      «что в работе». Кандидаты на `completed`/`rejected` по признакам (приоритет сверху вниз):
      (1) входит в спринт со `status: completed`/`rejected`, а сама всё ещё `active`;
      (2) на спеку извлечён ADR (спека фигурирует в `sources:` какого-либо ADR) — сильный сигнал закрытия;
      (3) git-история файла застыла > **~14 дней** (порог конфигурируемый) при статусе `active`.
      **Маркеры в ТЕЛЕ спеки — «ЗАКРЫТА», чек-боксы приёмки — НЕнадёжны** (расходятся с реальностью в обе
      стороны): приоритет спринт/ADR/git над телом; тело — вспомогательный сигнал, не основание.
      **Report-only, никогда не автофлип:** неверно закрыть живую спеку хуже дрейфа. Уметь предложить и
      `rejected` (отклонена гейтом), не только `completed`. При сомнении — не трогать, вынести человеку.
    ```
