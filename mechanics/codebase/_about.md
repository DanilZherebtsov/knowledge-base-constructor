# Механика: codebase

**Что делает (1 строка):** даёт классу **дом для собственного кода, который проект ведёт и деплоит** (лендинг, сайт, скрипты, бот) — корневая код-папка по сути, правило «Код побеждает вику», конвенции владения кодом (файлы на диске = источник правды, how-to в README, дисциплина секретов) и предложение пары ролей «правит продукт / деплоит» на спрос. Лёгкий слой, **не** цикл разработки (это `spec-lifecycle`) и **не** центральный тип вики.

## Слоты целевого проекта, которые трогает

- **OWNED-CODE** (в `methodology/ingest.md`, ветка привратника «рабочий код») — *заполняет.* Заменяет плейсхолдер на ссылку: раскладку код-папки, конвенции и предложение ролей ведёт `codebase.md`.
- **S4** (дополнительные рабочие слои для классов с кодом) — *объявляет* код-папку в дереве «Архитектура». В отличие от `spec-lifecycle` (раскрывает `specs/`+`src/`+`data/`) — здесь папка **названа по сути актива** (`landing/`, `src/`, `scripts/`), создаётся лениво.
- **S7** (правило авторитета) — *дополняет.* Класс без кода несёт «Источники побеждают вики»; `codebase` добавляет вторую половину для самого кода: «**код в код-папке побеждает вику** (для кода; знание о коде — в вике)». Класс с кодом получает обе половины.
- **S2** — *опирается* на наличие `decisions/` и `principles/` (решения о стеке/хостинге → ADR; уроки деплоя → principles).

## Файлы целевого проекта, которые трогает

- `methodology/codebase.md` — кладётся этот drop-in (основной файл механики).
- `roles/_разработчик.md`, `roles/_релиз-менеджер.md` — кладутся **только если активна механика `roles`** (образцы пары продуктовых ролей).
- `methodology/ingest.md` — заполнить слот OWNED-CODE (см. wiring).
- `CLAUDE.md` — три точки: строка код-папки в дереве «Архитектура» (слот S4); дополнение правила авторитета (слот S7); указатель `codebase.md` в разделе «Wiki: типы и операции».
- `methodology/bootstrap.md` — пункт в «Что НЕ создаётся при bootstrap»: код-папка не заводится пустой, появляется по решению человека через привратник.
- Создаёт во время работы (не при установке): код-папку `<asset>/` (по решению человека), `README.md` в ней, ADR в `decisions/`, записи в `STATE.md`.

## От каких механик зависит

- **base-скелет** — привратник в `ingest.md` (распознавание кода + слот OWNED-CODE), `wiki/{decisions,principles}/`, слоты S4/S7. Без привратника механике некуда зацепиться.
- **roles** (мягкая, опциональная) — нужна только для предложения ролей и образцов `_разработчик`/`_релиз-менеджер`. Нет `roles` — раздел «Роли продукта» в `codebase.md` и два образца не ставятся; остальное (код-папка, конвенции, секреты) работает.
- **lifecycle класса** (S6) — решения о стеке/хостинге роль/механика фиксирует как ADR через lifecycle класса (не решает за владельца).

## Несовместимость / композиция

- **С `spec-lifecycle` вместе не ставится.** spec-lifecycle уже несёт код-слой (`src/`/`data/`), цикл исполнения и authority «Код побеждает вику». `codebase` — облегчённая альтернатива для классов, которые ведут код, но не разрабатывают его из спек. В одном проекте — что-то одно.
- Центральный тип вики не занимает (в отличие от spec-/decision-/claim-): надстройка, а не основа. Совместима с любым lifecycle класса.

## Пошаговый wiring

1. **Скопировать drop-in:** `codebase.md` → `<target>/methodology/codebase.md`. Если активна `roles` — `roles/_разработчик.md` и `roles/_релиз-менеджер.md` → `<target>/roles/`.
2. **Заполнить слот OWNED-CODE** в `methodology/ingest.md` (ветка привратника «рабочий код»). Заменить плейсхолдер `<<СЛОТ OWNED-CODE: …>>` на: «раскладку код-папки, конвенции владения кодом и предложение ролей ведёт [codebase.md](codebase.md)».
3. **Объявить код-папку (слот S4).** В дереве «Архитектура» (`CLAUDE.md`) заменить `<<СЛОТ S4: …>>` (или добавить рядом с `output/`) на строку: «`<asset>/` ← Боевой код продукта (имя по сути: `landing/`, `src/`, `scripts/`). Источник правды на диске; ведётся и деплоится; создаётся по решению человека. Детали — [methodology/codebase.md](methodology/codebase.md).» Параллельно в разделе «Документы и нейминг» в строке «Куда класть» добавить: «боевой код → корневая код-папка (не `output/`, не `raw/`)».
4. **Дополнить слот S7** (правило авторитета в «Архитектура»). К строке авторитета класса добавить вторую половину: «**Код в код-папке побеждает вику** — для самого кода истина на диске; в вику идёт только знание о нём». (Класс без кода нёс только «Источники/Код побеждают вику» — теперь обе половины.)
5. **Указатель в «Wiki: типы страниц и операции» (always-on вход для обоих триггеров).** После доменного потока единицы работы добавить строку: «**Владение кодом** — код-папка (источник правды на диске), дисциплина секретов, роли продукта; **приходит готовый код ИЛИ решено делать код с нуля → сначала [methodology/codebase.md](methodology/codebase.md)**.» Это даёт триггеру «с нуля» (без ingest) always-on точку входа.
6. **Bootstrap.** В `methodology/bootstrap.md`, раздел «Что НЕ создаётся при bootstrap», добавить: «**Код-папка** — пустой не заводится; появляется, когда приходит рабочий код и человек подтверждает, что проект его ведёт (привратник в [ingest.md](ingest.md)).»
7. **Роли (если активна `roles`).** Никаких правок `roles.md` не требуется: образцы `_разработчик`/`_релиз-менеджер` подхватываются только через явное предложение в `codebase.md` (жёсткая привязка роль→образец); дефолтный путь `roles.md` → `_шаблон.md` для прочих ролей не меняется.
8. **Чего НЕ добавлять.** Не заводить код-папку пустой при bootstrap. Не дублировать код в `raw/`/`output/`. Не копировать сам код в `wiki/` (туда — только знание о нём). Не ставить вместе со `spec-lifecycle`.

## Деактивация (класс без собственного кода)

1. Не копировать `methodology/codebase.md` и образцы ролей.
2. Слот OWNED-CODE в `ingest.md` оставить в base-варианте («если проект всерьёз начинает владеть кодом — предложи подключить механику `codebase`»).
3. Слоты S4/S7 заполняются пресетом как обычно (S4 пусто или под `spec-lifecycle`; S7 — только авторитет класса).

## Подключить позже (в любой момент)

Проект, собранный без `codebase`, получает её в любой момент **без пересборки** — механика аддитивна, ничего эксклюзивного к моменту инициализации в ней нет.

1. **Триггер:** человек говорит «будем писать код» / «подключи работу с кодом»; ИЛИ привратник в `ingest.md` поймал входящий код и человек подтвердил владение; ИЛИ на вопрос про код при bootstrap ответили «да».
2. **Действие:** Claude выполняет wiring выше (шаги 1–7) против живого проекта — кладёт `codebase.md` (и образцы ролей, если активна `roles`), заполняет слоты OWNED-CODE / S4 / S7, добавляет указатели и пункт bootstrap.
3. **Источник файлов механики**, если леса конструктора уже убраны: апстрим-зеркало (тот же канал, что профилактика/lint использует для сверки версий по SHA). Локальные леса на месте — берём из них.
