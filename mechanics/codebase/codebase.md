# Codebase — проект владеет кодом

Подключается классам, которые **ведут и деплоят собственный код** (лендинг, сайт, скрипты, бот, приложение), но без спек-цикла. Если в проекте уже есть `spec-lifecycle` (разработка из спек) — эта механика не нужна: код-слой и дисциплину несёт он. `codebase` — лёгкий слой «тут живёт наш код и как с ним обращаться», не цикл разработки и не центральный тип вики.

## Код-папка — корневая, названа по сути

- Рабочий код живёт в **корневой папке, названной по тому, что это** (`landing/`, `src/`, `scripts/`, `bot/`) — **не** в `raw/` (там read-only архив) и **не** в `output/` (там расходные рабочие файлы).
- Папка создаётся **по решению человека** в момент, когда проект **впервые получает код** — двумя путями: *приходит готовый код* (привратник в [ingest.md](ingest.md)) **или** *решено делать код с нуля* (просьба «сделай сайт / бот / приложение», а код-папки ещё нет). Оба пути ведут в одну ветку: завести папку + (по гейту ниже) предложить роли. Пустой при инициализации (bootstrap) папка не создаётся.
- Несколько разнородных активов — несколько корневых папок (`landing/`, `bot/`), каждая по сути.

## Источник правды — файлы на диске

- **Код побеждает вику.** Для самого кода истина — файлы в код-папке, а не их пересказ в `wiki/`. Расхождение — верить диску.
- В `wiki/` идёт только **знание о коде**: что это, назначение/оффер, ключевые факты, принятые решения. Сам код, конфиги и runtime-файлы в вику не копируются.
- **How-to живёт в код-папке, не в вике:** как собрать, как задеплоить, что проверить перед запуском → `README.md` рядом с кодом. Вика — конспект знания, не тех-инструкция.
- Архитектурное/инфраструктурное **решение** (стек, хостинг, домен, способ деплоя) → ADR в `decisions/` через lifecycle класса. **Урок** («больше так не делаем») → `principles/`.

## Дисциплина секретов

- Секреты (`.env`, ключи, токены, конфиги с доступами) — только в файле под `.gitignore`. **Никогда** в `wiki/`, в git или в чат.
- Любой конфиг с доступами заносится в `.gitignore` **до** того, как туда впишут секрет.

## Изменения кода

- Точечные правки — правило «Точечные изменения» из `CLAUDE.md` действует и для кода.
- **Перед правкой общего файла перечитай его с диска:** его могли изменить в другом чате или роли. Правь поверх актуальной версии, не поверх копии из памяти.
- Деплой воспроизводим: одной командой, задокументирован в `README` код-папки.

## Роли продукта — предложение на спрос (гейт: деплоимый веб-продукт)

Две роли образуют пару «один правит — другой публикует». Эта ось есть **только у деплоимого веб-продукта** — сайт, веб-сервис, бот. Поэтому offer гейтится по типу кода:

- **Деплоимый веб-продукт** (сайт / веб-сервис / бот) — когда заводится код-папка (приходом кода или решением делать с нуля), **предложи** человеку две роли (по согласию, как любая роль — [roles.md](roles.md)):
  - **кто правит код продукта** → образец [roles/_разработчик.md](../roles/_разработчик.md);
  - **кто деплоит и эксплуатирует** → образец [roles/_релиз-менеджер.md](../roles/_релиз-менеджер.md).
- **Любой другой код** — мобильное/десктоп-приложение, библиотека, CLI, ML-модель, дата-пайплайн, разовый скрипт и т.п. — роли **не предлагать**. Дефолт гейта — **молчать**; offer срабатывает только для явного деплоимого веб-продукта выше. Пара «правит/деплоит» и образцы заточены под веб-выкладку (хостинг, домен, SSL, формы) и не подойдут иному коду. Код-папка и конвенции выше работают; роль — только на спрос обычным «создай роль» → `_шаблон.md`, если реально понадобится.

Согласился — запускается обычный поток «создай роль» ([roles.md](roles.md)), а **названный образец идёт как стартовый черновик-описание** (шаг 1 потока). Привязка роль→образец жёсткая — угадывания нет. Прочие роли (не эти две) идут по дефолту `roles.md` → `_шаблон.md`. Две роли работают в паре; протокол передачи — в самих образцах.

> Если механика `roles` в классе не подключена — этот раздел и образцы `_разработчик`/`_релиз-менеджер` не ставятся; код-папка и конвенции выше работают и без ролей.
