# Механика: question-lifecycle (S6 для research)

**Что делает (1 строка):** задаёт единицу работы research-класса — исследовательский вопрос Q-NNN — и его поток «повестка в STATE → рабочий файл в output → извлечение канона в wiki» без отдельной папки жизненного цикла.

## Слоты целевого проекта, которые трогает

- **S6** (доменный lifecycle-файл) — заполняется этой механикой: `question-lifecycle.md`.
- **S5** (тело state-rules) — механика наполняет тело `state-rules.md` research-семантикой (см. `state-rules-body.md` + шаг 6 wiring): 7 разделов, буфер «Отвечено», правило переноса вопроса при извлечении claim/synthesis.
- **S2** (центральный/доменный тип вики) — требует, чтобы среди типов был `claims/` (плюс `synthesis/`, `decisions/`, `principles/`), куда извлекается канон.

## Файлы, которые трогает

- `methodology/question-lifecycle.md` — кладётся (сама механика).
- `methodology/state-rules.md` — наполняется тело (S5-регион + правило 2б) из `state-rules-body.md`.
- `CLAUDE.md` (целевой) — добавляется указатель на S6 и проверяется пункт дисциплины «синтез при закрытии единицы работы».
- `STATE.md` (целевой) — рабочие файлы вопросов ссылаются из разделов 3–5 и 7.
- `wiki/claims/`, `wiki/synthesis/`, `wiki/decisions/`, `wiki/principles/`, `wiki/log.md` — целевые типы, куда уезжает извлечённый канон и запись `question-closed | Q-NNN`.
- `output/` — рабочий файл активного вопроса `q-NNN-<slug>.md`.

## От каких механик зависит

- **claim-graph** (обязательная) — статусы гипотез (`open`/`validated`/`invalidated`), supersession, claim-баланс. Ставится как ОТДЕЛЬНАЯ механика из `mechanics/claim-graph/`; здесь НЕ дублируется (контракт стыка: вопрос закрывается извлечением claim'ов, которыми управляет claim-graph).
- **roles** (ОПЦИОНАЛЬНАЯ) — подключается ТОЛЬКО если пресет перечислил `roles` в `mechanics`. Поток вопроса самодостаточен без ролей: human-in-loop подтверждение нетривиальных мутаций задаётся базовой дисциплиной `CLAUDE.md`, не требует `roles.md`. research роли не использует (ADR-0004) → пропустить.

## Пошаговый wiring

1. **Положить сам lifecycle-файл.** Скопировать `question-lifecycle.md` в `methodology/` целевого проекта. Это заполняет **S6**.
2. **Заполнить указатель S6 в CLAUDE.md.** В блоке «Wiki: типы страниц и операции» заменить плейсхолдер S6 на:
   `**Поток исследовательских вопросов** (повестка в STATE → рабочий файл в output → выжимка в wiki) — [methodology/question-lifecycle.md](methodology/question-lifecycle.md).`
   В дереве «Архитектура» строку S6 — на `question-lifecycle.md (поток вопроса: STATE → output → wiki)`.
3. **Подключить дисциплину закрытия.** Убедиться, что в разделе «Дисциплина» CLAUDE.md есть пункт «синтез знаний при закрытии единицы работы» и что «единица работы» определена как исследовательский вопрос со ссылкой на этот файл. (В base пункт уже есть со ссылкой на S6 — отдельно добавлять не нужно, только проверить, что S6 указывает сюда.)
4. **Обеспечить зависимость claim-graph (обязательную).** Подключить механику claim-graph (тип `wiki/claims/` + статусы + supersession) по её `_about.md`. Без неё поток вопроса не закрывается: извлечение канона уезжает в `claims/`.
5. **roles — ТОЛЬКО если пресет их включает.** research роли НЕ включает (в `mechanics` их нет) → шаг пропускается: `roles.md`/`roles/` не кладутся, раздел «Роли» в CLAUDE.md не добавляется. Если другой проект на этом lifecycle захочет роли — он добавит `roles` в свой манифест, и отработает wiring механики `roles`.
6. **Наполнить тело STATE (state-rules).** Вставить в base `state-rules.md` содержимое `state-rules-body.md` этой механики: заместить регион слота S5 («## Структура» с 7 research-разделами + гранулярность) и уточнить правило 2б (перенос вопроса «Сейчас в работе» → «Отвечено» при извлечении claim/synthesis). Базовые правила 1, 3–7 наследуются из base. Отдельной папки `output/questions/{...}/` НЕ заводить (сознательный отказ, см. lifecycle).
7. **Ничего лишнего не убирать из CLAUDE.md.** Механика заполняет S6 + тело state-rules и опирается на уже существующие разделы «Операционное состояние», «Дисциплина», «Wiki: типы».
