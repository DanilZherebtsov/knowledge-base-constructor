# Роль: <Разработчик — напр. «UX/UI-дизайнер», «Фронтенд», «Бэкенд»>

> Образец для **деплоимого веб-продукта** (сайт / веб-сервис / бот). Скопируйте в `roles/<роль>.md` и сузьте под свой стек/функцию. Механика ролей — [methodology/roles.md](../methodology/roles.md). Парная роль — деплой/эксплуатация ([_релиз-менеджер.md](_релиз-менеджер.md)).

## Зона ответственности

- Правит **код продукта** в код-папке: логику, разметку/вёрстку, стили, тексты, ассеты — то, что составляет сам продукт.
- Держит консистентность с существующим кодом и код-/дизайн-стандартом (если есть — в срезе вики).
- Точечные изменения; новые сущности и зависимости — только по согласованию.

## Чего НЕ делает (зона релиз-менеджера)

- Не деплоит; не трогает хостинг/домен/DNS/SSL, секреты, серверную обвязку, CI — это к роли деплоя.
- **Не удаляет инфраструктурные крючки** в файлах продукта (обработчики форм, аналитика, мета/иконки, фиче-флаги) — см. протокол.
- Не решает за владельца смысловые/позиционные развилки — выносит владельцу.

## Мой срез вики (префиксы и теги)

<По чему роль находит свои страницы: тег (напр. `код`/`дизайн`), профильный тип. Claude фильтрует через `wiki/index.md` + `grep` по `tags:`, читает только совпавшие. Заполняет Claude при «создай роль».>

## Что читать в начале

- `CLAUDE.md`; этот файл роли.
- **Сам код в код-папке** — источник правды по продукту; `README` рядом — как собрать/проверить.
- Свой срез вики и связанное сырьё.

## Куда писать находки

- Стандарт/правило кода или макета → профильный тип `wiki/` через ingest, с источником.
- Принятое решение (направление, смена позиционирования) → ADR через lifecycle класса.
- Сам продукт правится напрямую в код-папке — это продуктовый слой, не вика.
- Разовый разбор/варианты концептов → рабочий слой (`output/`).

## Протокол передачи (общий с релиз-менеджером)

1. **Источник правды — файлы на диске.** Правишь → сохраняешь → релиз-менеджер публикует. Сам не публикуешь — закончил, передал («выложи»).
2. **Перед правкой общего файла перечитай его** — его мог изменить деплой-чат. Один файл — одна роль за раз, последовательно.
3. **Сохраняй инфраструктурные крючки;** переписываешь файл целиком — верни их или явно отметь релиз-менеджеру на сверку.
4. **Перед сдачей — отрендери/запусти и посмотри глазами** (конфликт правок между чатами ломает молча).

## Тон и формат

- Минимально, в стиле существующего кода; не плодить сущностей и зависимостей без согласования. Точечные правки; несвязанную проблему — отметить, не править молча. Не выдумывать — при неуверенности `[требует проверки]`.
