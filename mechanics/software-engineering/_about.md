# Механика: software-engineering

**Что делает (1 строка):** даёт классу **полную компетенцию работы со своим кодом** — дом для кода (корневая код-папка по сути), правило «код побеждает вику», **цикл исполнения** (постановка → гейт независимой проверки субагентами → реализация → повторный гейт прогоном тестов → доклад; петля на ошибках; эскалация блокеров), дисциплину секретов и runtime-данных, воспроизводимый деплой и предложение пары ролей «правит/деплоит» на спрос. Пишет код **с нуля или поверх существующего**. Центральный тип вики **не** занимает; композируется с любым lifecycle класса (в паре со `spec-lifecycle` у saas: спека — единица работы, эта механика — исполнение).

## Слоты целевого проекта, которые трогает

- **OWNED-CODE** (в `methodology/ingest.md`, ветка привратника «рабочий код») — *заполняет.* Заменяет плейсхолдер на ссылку: раскладку код-папки, конвенции, цикл и предложение ролей ведёт `software-engineering.md`.
- **S4** (рабочие слои для классов с кодом) — *объявляет* код-папку (названа по сути актива: `landing/`, `src/`, `scripts/`) + ленивые `src/`/`data/`/`scripts/`. Папка `specs/` — не отсюда: её объявляет `spec-lifecycle`, если класс его ведёт.
- **S7** (правило авторитета) — *дополняет.* К строке авторитета класса добавляет «**код в код-папке побеждает вику** — для самого кода истина на диске; в вику идёт только знание о нём».
- **always-on цикл** (в `CLAUDE.md`, «Принципы работы Claude → фаза действуй») — *вставляет* строку-указатель на цикл исполнения, иначе протокол читался бы только по триггеру.
- **S2** — *опирается* на наличие `decisions/` и `principles/` (решения о стеке/хостинге → ADR; уроки деплоя → principles).

## Файлы целевого проекта, которые трогает

- `methodology/software-engineering.md` — кладётся этот drop-in (основной файл механики).
- `roles/_разработчик.md`, `roles/_релиз-менеджер.md` — образцы пары продуктовых ролей; `roles/` есть всегда (базовая машинерия, ADR-0027), образцы предлагаются по гейту деплоимого веб-продукта.
- `methodology/ingest.md` — заполнить слот OWNED-CODE (см. wiring).
- `CLAUDE.md` — четыре точки: строка код-папки в дереве «Архитектура» (S4); дополнение правила авторитета (S7); always-on строка про цикл исполнения («фаза действуй»); указатель `software-engineering.md` в разделе «Wiki: типы и операции». Плюс правило про `data/` в «Дисциплине», если work-layers включает `data/`.
- `methodology/bootstrap.md` — код-папка пустой не заводится; `src/`/`data/` ленивые.
- Создаёт во время работы (не при установке): код-папку `<asset>/` (по решению человека), `README.md` в ней, ADR в `decisions/`, записи в `STATE.md`.

## От каких механик зависит

- **base-скелет** — привратник в `ingest.md` (распознавание кода + слот OWNED-CODE), `wiki/{decisions,principles}/`, слоты S4/S7, точка инъекции цикла в «фаза действуй». Без них механике некуда зацепиться.
- **roles** — **базовая машинерия, есть всегда** (ADR-0027, не механика-зависимость). Образцы `_разработчик`/`_релиз-менеджер` кладутся с этой механикой в `roles/`; сам раздел «Роли продукта» предлагает их по гейту деплоимого веб-продукта.
- **lifecycle класса** (S6) — решения о стеке/хостинге фиксируются как ADR через lifecycle класса (механика не решает за владельца). У saas работает в паре со `spec-lifecycle` (вход цикла = спека); без него вход цикла = постановка в `STATE.md`/чате.

## Композиция

- **Со `spec-lifecycle` — композируется, не конфликтует.** Это две оси: `spec-lifecycle` — единица работы (спека/спринт, статусы, ADR-on-close, синтез), `software-engineering` — компетенция кода (цикл, владение, деплой). У saas обе: спека даёт что/как-отслеживаем, эта механика — как код пишется и проверяется. Эксклюзивности нет.
- **Центральный тип вики не занимает** (в отличие от spec-/decision-/claim-): надстройка, а не основа. Совместима с любым lifecycle класса.

## Пошаговый wiring

1. **Скопировать drop-in:** `software-engineering.md` → `<target>/methodology/software-engineering.md`. Образцы `roles/_разработчик.md` и `roles/_релиз-менеджер.md` → `<target>/roles/` (папка `roles/` есть всегда — базовая машинерия).
2. **Заполнить слот OWNED-CODE** в `methodology/ingest.md` (ветка привратника «рабочий код»). Заменить плейсхолдер `<<СЛОТ OWNED-CODE: …>>` на: «раскладку код-папки, конвенции владения кодом, цикл исполнения и предложение ролей ведёт [software-engineering.md](software-engineering.md)».
3. **Объявить код-папку (слот S4).** В дереве «Архитектура» (`CLAUDE.md`) заменить `<<СЛОТ S4: …>>` (или добавить рядом с `output/`) на строку: «`<asset>/` ← Боевой код продукта (имя по сути: `landing/`, `src/`, `scripts/`). Источник правды на диске; ведётся и деплоится; создаётся по решению человека. Детали — [methodology/software-engineering.md](methodology/software-engineering.md).» Если класс ведёт `spec-lifecycle` — рядом стоит и `specs/` (его объявляет тот). Параллельно в «Документы и нейминг → Куда класть» добавить: «боевой код → корневая код-папка (не `output/`, не `raw/`)».
4. **Дополнить слот S7** (правило авторитета). К строке авторитета класса добавить вторую половину: «**Код в код-папке побеждает вику** — для самого кода истина на диске; в вику идёт только знание о нём».
5. **Always-on указатель на цикл.** В «Принципы работы Claude → В процессе — фаза действуй» добавить строку (после п.4 «Выполнение от цели»): «**Код — только через цикл исполнения**: постановка → независимая проверка субагентами (3 на корректность + 1 на влияние на модули) → реализация → повторная проверка прогоном тестов → доклад; ошибка → петля по циклу; пробел в скоупе → в постановку; блокер вне скоупа → к человеку с вариантами. Полностью — [methodology/software-engineering.md](methodology/software-engineering.md).» Без этого протокол читается только «по триггеру», и Claude может начать кодить, не открыв файл механики.
6. **Указатель в «Wiki: типы страниц и операции» (always-on вход для обоих триггеров).** После доменного потока единицы работы добавить строку: «**Владение кодом и цикл исполнения** — код-папка (источник правды на диске), дисциплина секретов, цикл написания кода, роли продукта; **приходит готовый код ИЛИ решено делать код с нуля → сначала [methodology/software-engineering.md](methodology/software-engineering.md)**.» Это даёт триггеру «с нуля» (без ingest) always-on точку входа.
7. **Runtime-данные (только если work-layers включает `data/`).** Вставить в раздел «Дисциплина» CLAUDE.md, перед правилом «Синтез знаний», правило: «**Runtime-данные — в `data/`, не в `wiki/`.** Промпты, KB-словари, конфиг-файлы, шаблоны лежат в top-level `data/`; их потребляет рантайм, Claude не читает их как знание. В `wiki/` идёт только **контракт** (схема + способ обновления). Положить runtime-файл в `wiki/` — типовая ошибка.»
8. **Bootstrap.** В `methodology/bootstrap.md`, раздел «Что НЕ создаётся при bootstrap», добавить: «**Код-папка** — пустой не заводится; появляется, когда приходит рабочий код или решено делать код с нуля, и человек подтверждает (привратник в [ingest.md](ingest.md)). `src/`/`data/`/`scripts/` тоже не создаются пустыми — по необходимости.»
9. **Роли (всегда есть — базовые).** Никаких правок `roles.md` не требуется: образцы `_разработчик`/`_релиз-менеджер` подхватываются только через явное предложение в `software-engineering.md` (жёсткая привязка роль→образец); дефолтный путь `roles.md` → `_шаблон.md` для прочих ролей не меняется.
10. **Чего НЕ добавлять.** Не заводить код-папку пустой при bootstrap. Не дублировать код в `raw/`/`output/`. Не копировать сам код в `wiki/` (туда — только знание о нём). Не дублировать цикл в lifecycle-файле класса — он живёт здесь, lifecycle лишь ссылается.

## Деактивация (класс без собственного кода)

1. Не копировать `methodology/software-engineering.md` и образцы ролей.
2. Слот OWNED-CODE в `ingest.md` оставить в base-варианте («если проект всерьёз начинает владеть кодом — предложи подключить механику `software-engineering`»).
3. Слоты S4/S7 заполняются пресетом как обычно (S4 пусто или под `spec-lifecycle`; S7 — только авторитет класса). Строку про цикл в «фаза действуй» и указатель в «Wiki» не вставлять.

## Подключить позже (в любой момент)

Проект, собранный без `software-engineering`, получает её в любой момент **без пересборки** — механика аддитивна, ничего эксклюзивного к моменту инициализации в ней нет.

1. **Триггер:** человек говорит «будем писать код» / «подключи работу с кодом»; ИЛИ привратник в `ingest.md` поймал входящий код и человек подтвердил владение; ИЛИ на вопрос про код при bootstrap ответили «да».
2. **Действие:** Claude выполняет wiring выше (шаги 1–9) против живого проекта — кладёт `software-engineering.md` (и образцы продуктовых ролей в `roles/` — папка есть всегда, базовая), заполняет слоты OWNED-CODE / S4 / S7, добавляет always-on строку цикла и указатели, пункт bootstrap.
3. **Источник файлов механики**, если леса конструктора уже убраны: апстрим-зеркало (тот же канал, что профилактика/lint использует для сверки версий по SHA). Локальные леса на месте — берём из них.
