# Сборка проекта из конструктора

Эту процедуру выполняет Claude в **чистой комнате**: видит ТОЛЬКО эту папку (`base/`, `mechanics/`, `presets/`, каталог). Папку `templates/` НЕ читает — иначе тест ложно пройдёт (дозаполнит пробелы по памяти).

## Вход

Обычный вход — разговорный, через [START-HERE.md](START-HERE.md): интервью без обязательного называния класса определяет стартовый пресет и доп-механики. Этот файл (ASSEMBLY) — механика самой сборки, на которую START-HERE опирается (шаг 4). Прямой вход «имя пресета» (`saas-product` / …) — режим детерминированной сборки/теста.

## Процедура

> **Язык разговора — сквозной.** Всё, что видит человек в ходе сборки, включая служебные реплики о ходе работ, — на его языке и без внутреннего жаргона. Правило и стоп-слова — [START-HERE.md](START-HERE.md), «Язык разговора»; шаг 9 ниже — частный случай, а не единственное место, где запрет действует (ADR-0030).

1. **Старт от base.** Скопировать `base/` как заготовку проекта (скелет со слотами `<<СЛОТ ...>>`).
2. **Прочитать манифест** `presets/<class>.md`.
3. **Подставить скалярные слоты** из манифеста в base:
   - `title-word` → слот `TITLE` в H1 + проза вводных абзацев; `S1` «О проекте» — из интервью (шаг 7).
   - **отпечаток сборки (provenance)** → слот `PROVENANCE` в `CLAUDE.md` (строка 3). Прочитать `versions.json` конструктора и записать строку из РЕАЛЬНО ВШИТЫХ частей: `> **Сборка:** <class> · base@<v> · <каждая включённая механика>@<v> · <манифест класса>@<v>. На lint Claude сверяет части с апстрим-`versions.json` — [methodology/lint.md](methodology/lint.md).` Пример (saas): `> **Сборка:** saas-product · base@24 · spec-lifecycle@4 · software-engineering@3 · saas-product@2.` Роли в отпечаток НЕ пишутся — это базовая машинерия, не механика (ADR-0027). Не подключённые опциональные механики НЕ пишутся. Это единственное, что нужно проекту для обновлений; CHANGELOG/полную историю проект не несёт.
   - `central-type` (S2) → дерево `wiki/` + описание типа; `S3` → `page-conventions.md`; `work-layers` (S4) → дерево; `state-sections` (S5) → `STATE.md` + `state-rules.md`; `authority` (S7); `domain-conv` (S8); `INTERVIEW-Q`+`raw-defaults` → `bootstrap.md`; `domain-lint` → `lint.md`; `close-op` → `index-log-format.md`; `KNOWLEDGE-UNIT` → `ingest.md`.
   - Слоты с пометкой «иначе пусто», которые не заполняет ни одна подключённая механика (`DEADLINE-CHECK` у класса без `decision-lifecycle`) → удалить маркер (резолв в пусто).
4. **Вшить механики** из `manifest.mechanics` по их `mechanics/<имя>/_about.md` (раздел «Пошаговый wiring») + обязательные зависимости (`claim-graph` для `question-lifecycle`).
   - **Конфликт за центральный тип разрешать самому, молча** (ADR-0030). Претендуют двое (`claim-graph` / `spec-lifecycle` / `decision-lifecycle`) — победитель определяется **критерием** в [mechanics-catalog.md](mechanics-catalog.md) («Критерий центрального типа»): пересматривает ли проект собственные выводы (→ `claim-graph`) или действует и идёт дальше (→ lifecycle класса). Проигравший не подключается. **Человеку конфликт не выносить** и внутренними словами не описывать — у него нет базы для такого выбора. Наружу — только последствие на его языке, и только если оно у него есть.
5. **Базовая машинерия не деактивируется.** Роли (`roles.md`, `roles/_шаблон.md`, раздел «Роли», строка bootstrap) присутствуют всегда — из base (ADR-0027). Заполнить слот `ROLES-FILL` примерами ролей класса. Опциональных механик, требующих выкорчёвывания из base-скелета, сейчас нет.
6. **Состав выхода — что идёт в проект.** В собранный проект идут ТОЛЬКО: `CLAUDE.md`, `STATE.md`, `methodology/*.md` (классовый набор), `roles/` (всегда — базовая машинерия ролей), рабочие слои по `work-layers` (`specs/` и т.д., с учётом ленивого создания — `spec-lifecycle`). **НЕ идут:** `base/README.md`, `CHANGELOG.md` (единый, в корне конструктора) и никакие внутренние файлы конструктора (`ASSEMBLY.md`, `EXPERIMENT.md`, `mechanics-catalog.md`, `mechanics/`, `presets/`). **CHANGELOG — артефакт версионирования репозитория шаблонов** (его ведёт операция «пересобрать пресеты / распространить base», фиксируя, что изменилось), а НЕ продукт пер-проектной сборки. Проект несёт только строку-метку версии (шаг 3).
7. **Интервью с пользователем** (bootstrap шаг 1): instance-данные — «О проекте» (S1), реальные источники. Заполнить, удалить инструктивные сноски и base-врезку «Это base-template — скелет».
8. **Гигиена.** `grep -rn '<<СЛОТ' .` и `grep -rn 'ROLES-FILL' .` — пусто (литералы `<<СЛОТ …>>` внутри base-прозы `lint.md` — не маркеры, а текст инструкции). Нет битых ссылок на не подключённые опциональные механики.
9. **Онбординг (финальный хэндофф).** После сборки и уборки лесов дай человеку короткую ориентацию — **простым языком, веди с фразы-действия, а не с термина** (никаких «ingest / query / lint» первым словом; внутренний жаргон — механики, центральный тип, слои, версии — в реплике пользователю НЕ употреблять):
   - **Куда кидать материалы** — «Новые документы, заметки, ссылки кидай в папку `input/`, потом скажи „обработай“ — я сам разложу по местам и усвою» (подпапки человеку знать не нужно).
   - **Как пользоваться** (перечисляй ТОЛЬКО то, что реально собрано):
     - «**обработай это** / запомни / добавь в память» — разбираю материал и кладу в память проекта;
     - «**что мы знаем про X** / сравни A и B» — отвечаю из памяти, со ссылками на источники;
     - «**сделай профилактику**» — проверяю целостность памяти + есть ли обновления конструктора (+ близкие/просроченные обязательства, если собран их календарь);
     - *поток работы класса* — своей человеческой фразой («взять задачу в работу» / «ответить на вопрос» / «принять решение»), если он собран;
     - «**создай роль …**» / «**работаем в роли …**» — доступно всегда (роли есть в любом проекте); по желанию заранее закинуть в `input/` описание роли и материалы — создастся сразу квалифицированно;
     - особые запросы (обратный поиск по источникам, карта свидетельств) — только если собран claim-граф.
   - **Заметки (текущее состояние)** — стадия; что ещё не создано и появится по факту (`src/`/`data/` — при появлении кода/данных); под git проект или нет.

## Результат

Содержимое проекта (`CLAUDE.md` + `methodology/` + `STATE.md` + рабочие слои), функционально эквивалентное `templates/<class>/`. Совпадать побайтово не обязано — косметический пересказ base-прозы роняем сознательно; обязана присутствовать каждая реальная доработка из `customizations_checklist` манифеста. `CHANGELOG.md`/`README.md` — вне сборки (репо-/версионный слой).
