# Миграция 0002 — роли в base (неотключаемая машинерия)

> **Исполняется агентом ВНУТРИ боевого репозитория инстанса.** Подходит любому собранному проекту, чей отпечаток несёт `base@≤23`. Источник base-скелета — публичное зеркало `knowledge-base-constructor` (тот же канал, что профилактика использует для сверки версий). Решение — ADR-0027.

## Что меняется и главный инвариант

Машинерия ролей (`methodology/roles.md`, `roles/_шаблон.md`, раздел «## Роли» в `CLAUDE.md`, строка в `bootstrap.md`) была отключаемой (ветка «Деактивировать» механики `roles`) — и на части инстансов оказалась **выкорчевана**, из-за чего «создай роль» не работал. Теперь это **неотключаемая часть base**: доустанавливается, если отсутствует; механика `roles` ретайрнута, метка `roles@N` из отпечатка убирается.

> **Инвариант миграции.** Меняется **только СТАНДАРТ-слой** — `methodology/roles.md`, каркас-проза `CLAUDE.md` (раздел «## Роли» + дерево), `bootstrap.md`, отпечаток сборки; плюс появляется `roles/_шаблон.md`. Всё остальное — байт-в-байт: `wiki/` (тела страниц, доп-типы), `raw/`, `STATE.md`, `output/`, `specs/`, `src/`, `data/`. Проверяется diff'ом (Шаг 5).

## Шаг 1. Определить входное состояние

Прочитать отпечаток (строка 3 `CLAUDE.md`) и проверить наличие ролей на диске:
- **Ветка A — роли ВЫКОРЧЕВАНЫ:** нет `methodology/roles.md`, нет папки `roles/`, нет раздела «## Роли» в `CLAUDE.md`. (Отпечаток `roles` не несёт.) → Шаг 2.
- **Ветка B — роли ЕСТЬ** (`roles@1` в отпечатке, файлы на месте). → пропустить Шаг 2, сразу Шаг 3.

## Шаг 2. Доустановить машинерию ролей (только ветка A)

Из зеркала `knowledge-base-constructor` (`constructor/base/`):
1. `base/methodology/roles.md` → `methodology/roles.md`. Заполнить слот `<<СЛОТ ROLES-FILL: …>>` примерами ролей своего класса (business — юрист/финансист/…; продукт с кодом — devops/security-reviewer/support; research — методолог/фактчекер/рецензент). Если ведётся lifecycle-класс — оставить ссылку на его файл в «Куда писать находки»; нет типа `decisions/` — снять ссылку на ADR.
2. `base/roles/_шаблон.md` → `roles/_шаблон.md` (создать папку `roles/`).
3. В `CLAUDE.md` добавить раздел «## Роли (специализированные чаты-сотрудники)» (между «## Wiki: типы страниц и операции» и «## Дисциплина») по base@24 — с аффордансом `input/` (заранее закинуть описание роли и материалы → создастся квалифицированно).
4. В дереве «Архитектура» (`CLAUDE.md`): добавить `roles.md` в блок `methodology/` и top-level строку `roles/`.
5. В `methodology/bootstrap.md`, «Что НЕ создаётся при bootstrap» — добавить строку про роли (base@24, с аффордансом `input/`).

## Шаг 3. Обновить отпечаток сборки

Строка 3 `CLAUDE.md`: `base@N` → `base@24`; **убрать `roles@1`**, если он есть (роли больше не версионируемая часть — они всегда в base). Прочие части не трогать.

## Шаг 4. Доклад человеку — нормальным языком

Коротко и без жаргона: «Обновил заготовку: теперь помощника под функцию можно завести в любой момент — „создай роль <название>“; по желанию заранее кинь в `input/` описание роли и материалы, тогда она соберётся сразу с контекстом.» Не перечислять внутренние файлы/версии.

## Шаг 5. Проверка

- `grep -rn '<<СЛОТ' .` и `grep -rn 'ROLES-FILL' .` — пусто.
- `roles/_шаблон.md` и `methodology/roles.md` на месте; раздел «## Роли» в `CLAUDE.md` есть; строка `roles@…` из отпечатка убрана, `base@24`.
- Diff подтверждает: `wiki/`, `raw/`, `STATE.md`, рабочие слои — не тронуты.
