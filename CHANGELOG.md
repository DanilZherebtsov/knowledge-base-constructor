# CHANGELOG — конструктор

Единый журнал версий всех частей конструктора. Версии частей — в [versions.json](versions.json); каждая часть (base, механики) версионируется **независимо**: правка только одной механики поднимает только её номер, `base` при этом не растёт. Собранные проекты ловят апдейты на профилактике по отпечатку (`base@N · <механики>@N`).

**Один файл, записи помечены частью.** Заголовок записи — `## <часть>@<версия>` (напр. `## base@19`, `## spec-lifecycle@2`). Сгруппировано по части, новейшая версия сверху. Отдельных changelog'ов у механик нет — всё здесь. (Пресеты-манифесты тут не ведутся: их «changelog» — сам файл `presets/<класс>.md`, профилактика читает его напрямую.)

**Дисциплина (для разработчика конструктора).** **Поднял версию части в `versions.json` — в ТОТ ЖЕ коммит впиши сюда запись `## <часть>@<версия>`.** Без неё downstream-проект ловит на профилактике бамп, который не может ни классифицировать, ни объяснить человеку (реальные кейсы: base@18 и spec-lifecycle@2 уехали без записей — профилактика верно отказалась применять их вслепую). Привратник `publish-constructor.yml` → `.github/scripts/check_changelogs.py` блокирует публикацию зеркала, если у части номер в `versions.json` и верхняя запись здесь разошлись.

**Что в записи (иначе бесполезна для триажа на той стороне):** метка влияния — `[инструкция]` (проза/правило, знания вики не трогаются) или `[формат]` (новое поле/раздел на страницах вики → нужна миграция, дописать «как принять»: что мигрировать, lazy/eager/skip); и **конкретная суть** — какие файлы/правила изменены и зачем, в терминах, по которым downstream сам классифицирует радиус и переведёт человеку. Не «выровнено по vN»: скупая запись прячет правки.

## base@20
`[инструкция]` В `CLAUDE.md` (раздел «Wiki: типы страниц и операции») — always-on строка-ловец «сделать сайт/бота/скрипт/приложение с нуля»: это работа со своим кодом, не общий консалтинг. Подключена механика кода (`codebase`/`spec-lifecycle`) — вести по ней; не подключена — не уходить по умолчанию в советы про внешние no-code-конструкторы, а развести две трактовки (свой код → предложить подключить `codebase` + роли «правит/деплоит»; либо no-code/делегируем вовне → остаёмся без кода) и зафиксировать выбор. Закрывает дыру: триггер «делать код с нуля» жил только внутри `codebase.md`, поэтому проект, собранный без этой механики, на первую же просьбу «сделай сайт» молча уходил в общий консалтинг и не предлагал ни механику, ни роли (реальный кейс: проект собран как «кода не будет», первая задача — сайт, агент предложил no-code-конструктор без разведения трактовок). Триггер always-on → ловит на любом этапе, не только при сборке; ненадёжный ранний ответ интервью «будет ли код» больше не вредит. Знания не трогаются.

## base@19

## base@18

Закрывает кейс боевого инстанса: готовый лендинг не распознался как живой код — ingest унёс сайт в `raw/`, агент предложил вести его в `output/`. Знания не трогаются.

## base@15
`[инструкция]` Flow B: проверку версий читать **по фиксации коммита, а не по ветке `main`** — `git ls-remote` за актуальным HEAD, затем raw-URL с этим SHA в пути. Отменяет ошибочную посылку v11, будто contents API «свежий без кэша»: и contents API, и `raw/main` — независимые пограничные кэши, каждый может отстать, поэтому два чтения одной ветки дают расходящиеся числа без способа выбрать верное (реальный кейс: API отдал 12, raw — 8 при истинных 14). URL с SHA адресуется содержимым — кэш не вернёт чужие байты, расхождение «какому чтению верить» исчезает по построению. Знания не трогаются.

## spec-lifecycle@3
`[инструкция]` Усиление цикла исполнения (боевой кейс: слишком широкое гейт-условие → ложные металл-коды → необратимая тихая порча калькулятора — дефект жил на уровне плана, ценность дал гейт ДО кода). Четыре доработки: **(R1)** спринт-спека/план проходит независимый гейт ДО старта задач (декомпозиция, ширина условий и допущений дизайна, совокупное влияние) — раньше гейтились только task-спеки, дефекты плана проскакивали; **(R2)** пост-гейт после кода проверяет корректность **исполнением** — прогнать/написать тесты, воспроизвести поведение, не только ревью кода агентами; **(R3)** гейт масштабируется под риск — impact-ревьюер помечает необратимость/блласт-радиус, необратимые/широкие изменения получают больше верификаторов + отдельный adversarial-раунд, рутинные — базовые 3+1; **(R4)** заметка для человека в тексте механики: называть задачу достаточно, цикл обязателен по умолчанию. Знание — ADR-0016 (раздел «Доработки»). Знания вики не трогаются.
