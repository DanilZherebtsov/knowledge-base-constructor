# Lint — проверка здоровья вики

Запускается фразами: «прогони lint», «проверь вики», «проверь целостность». Стартовый триггер свежести (когда сам предложить прогон) — в CLAUDE.md, §5 «Дисциплины».

Периодическая проверка. Два уровня с разной степенью полномочий.

## Автоисправление (детерминированное — Claude правит сам)

- Страницы, отсутствующие в `wiki/index.md` → добавить с заглушкой `(no summary)`.
- Записи индекса на удалённые файлы → пометить `[MISSING]` (не удалять).
- Битые внутренние ссылки с единственным одноимённым файлом в другом месте → починить путь.
- Ссылки в `sources:` на перемещённые сырые файлы с единственным кандидатом → починить.
- Отсутствующие взаимные ссылки между очевидно связанными страницами → добавить.

## Только отчёт (эвристическое — требует суждения человека)

- Страницы-сироты без входящих ссылок.
- Концепции, упомянутые на нескольких страницах, но без собственной страницы.
- Устаревшие утверждения, противоречащие более новому источнику.
- ADR со статусом `active`, противоречащие более новому ADR.
- Страницы больше 800 строк.
- Страницы без поля `sources:`.
- **Нарушение глубины:** любой `.md` в `wiki/<type>/<subdir>/...` (страницы должны быть плоско в type-folder). Исключение — `wiki/index.md` и `wiki/log.md`.
- **Лог не по возрастанию дат:** в `wiki/log.md` дата записи раньше даты предыдущей → сообщить (вероятно, дата взята с mtime файла или с даты работы, а не из окружения). Не сортировать автоматически — что чинить, штамп даты или позицию записи, решает человек.

**Доменные проверки центрального типа.** <<СЛОТ DOMAIN-LINT: класс с центральным типом добавляет сюда свою проверку. Примеры: `architecture/` без поля `implementation:` или с битыми путями (code drift, saas); `claim` со статусом active/validated, чьи counter-evidence перевешивают, и цитата без локализации (research); просроченные обязательства из `STATE.md` «Календарь обязательств» (business).>>

## Проверка обновлений (по отпечатку сборки)

Проект собран из конструктора и несёт **отпечаток** в строке 3 `CLAUDE.md`: из каких версий частей он собран (`base@N · <механики>@N · <манифест>@N`).

**Этот блок — БЕЗУСЛОВНЫЙ: выполняется целиком на КАЖДОМ прогоне lint, независимо от состояния вики.** Пуста ли вика, только что развёрнута через bootstrap, не менялась с прошлого раза, N-й lint подряд, health чист — в условие этой проверки не входит ВООБЩЕ: апстрим (конструктор) развивается отдельно от твоей вики. Отговорки «ничего не менялось», «можно не гонять», «уже проверял», «вика та же» — запрещены; это и есть тот провал, ради которого блок сделан обязательным.

**Без двух артефактов проверка НЕ выполнена** — не выводи вердикт по обновлениям и не дописывай `wiki/log.md`, пока в твоём ответе нет: (а) вставленного тела ответа `versions.json`; (б) таблицы сравнения по частям с числами. «Перепроверил, обновлений нет» без них — пустое утверждение, а не результат.

1. **Прочитать свой отпечаток** (строка 3 `CLAUDE.md`) — список `часть@версия`.
2. **Запросить свежие версии реальной командой (не из памяти/прошлого вывода) и ВСТАВИТЬ её вывод в ответ.** Читать **по фиксации коммита, не по ветке** — два шага:
   ```bash
   SHA=$(git ls-remote https://github.com/DanilZherebtsov/knowledge-base-constructor.git HEAD | cut -f1)
   curl -s "https://raw.githubusercontent.com/DanilZherebtsov/knowledge-base-constructor/$SHA/versions.json"
   ```
   **Почему так, а не по `main`:** и `raw.githubusercontent/main`, и GitHub contents API — оба пограничные кэши; каждый может независимо отстать, поэтому два чтения одной ветки дают расходящиеся числа и выбрать верное нельзя (был реальный случай: contents API отдал 12, raw — 8 при истинных 14). `git ls-remote` берёт текущий HEAD из git-протокола (мимо raw-CDN), а URL с SHA в пути **адресуется содержимым** — никакой кэш физически не вернёт под этот SHA чужие байты. Снимок гарантированно цельный и актуальный — расхождение «какому чтению верить» исчезает по построению. Нет `git`/`curl` — web-запросом: сперва `…/commits/main` (взять поле `sha`), затем тот же raw-URL с этим SHA в пути. SHA-фиксация важнее свежести самого SHA: даже слегка устаревший коммит даёт согласованный снимок, а не смесь чисел из разных кэшей. Совсем недоступно → так и сказать, не выдумывать «совпадает».
3. **Сравнить по частям и показать таблицу** `часть | у меня | апстрим | вердикт`. Апстрим выше моего → у части апдейт. Части, которых у проекта НЕТ (невшитые механики), — игнорировать. Вердикт «обновлений нет» допустим, ТОЛЬКО если в показанной таблице у каждой части `у меня == апстрим`. Сверяются ТОЛЬКО версии частей, не файлы целиком — косметика и инстанс-контент в сравнение не попадают.
4. **Для каждой обновившейся части — разобраться самому** (внутренняя оценка, не вывод человеку): прочитать её changelog и контент в конструкторе (`base/CHANGELOG.md`; `mechanics/<имя>/CHANGELOG.md`; для манифеста — `presets/<класс>.md`). Классифицировать радиус: *`[инструкция]`* (проза/правило) — правится файлами, знания (`wiki/`) не трогаются; *`[формат]`* (новое поле/раздел на страницах) — требует миграции `wiki/`.
5. **Доложить человеку — простым языком, без IT-жаргона.** Changelog написан для разработчика — не пересказывай дословно, переведи в суть. На каждую обновившуюся часть человек должен понять три вещи:
   - **Что изменилось** — 1–2 фразы в терминах пользы, не механики.
   - **Заденет ли его** — либо «трогать ничего не надо, это про то, как я сам работаю» (`[инструкция]`), либо «у заметок появится <что именно> — можно дописать не сразу» (`[формат]`).
   - **Брать или нет** — рекомендация одной фразой.

   В реплике человеку НЕ употреблять: «отпечаток / версия / base@N / механика / манифест / миграция / lazy-eager-skip / `[формат]`-`[инструкция]` / радиус / API».
   > ✗ Жаргон: «base@12: `[инструкция]` Flow B теперь через contents API, кэш raw устранён, миграция не нужна.»
   > ✓ По-человечески: «Подтянул улучшение в том, как я проверяю обновления: раньше иногда сверялся с чуть устаревшим списком и мог проморгать свежий апдейт — теперь всегда смотрю на самый актуальный. Твоих записей не касается, делать ничего не нужно. Советую принять — просто надёжнее.»

   Затем **предложить применить**: `[инструкция]` — точечно, с подтверждения; `[формат]` — назвать по-человечески объём («поле добавится примерно в N заметок») и выбор: **(а)** дописывать по ходу, когда открываем заметку, — по умолчанию; **(б)** пройтись по всем разом сейчас; **(в)** пока отложить. Записи массово не переписывать без согласия. Применили — **поднять версию этой части в отпечатке** (строка 3).

После lint — дописать в `wiki/log.md`:
```
## [YYYY-MM-DD] lint | найдено N проблем, исправлено M
```
