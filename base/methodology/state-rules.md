# STATE.md — структура и правила обновления

`STATE.md` в корне — единственное место для текущих планов и прогресса. Wiki хранит знания (append-mostly, с источниками, атомарно); STATE хранит намерения (частые перезаписи, без источников, всё в одном файле для обзорности).

## Структура

Фиксированный набор разделов, всегда в одном порядке. Пустые — с пометкой `_пусто_`, не удаляются (глаз привыкает к расположению).

Класс может переименовать/переосмыслить (пример: business — Профиль / Открытые направления / На этой неделе / Следующее / Календарь обязательств / Блокеры,риски,решения / Недавно завершено).>>

**Гранулярность отражает горизонт.** В начале — крупными мазками; по мере приближения дробится. Не атомизировать преждевременно (ложная точность).

## Правила обновления

1. **Источник правды.** STATE — единственное место для текущего плана. Никаких параллельных списков.
2. **Claude обновляет в трёх моментах:** (а) молча читает в начале сессии; (б) после завершения пункта переносит его в буфер «завершено»; (в) после ingest, меняющего приоритеты, — предлагает правку, человек подтверждает.
3. **Человек обновляет в двух:** смена приоритетов; новый блокер. Через чат или руками.
4. **Свежесть по дате.** Поле `_Обновлено:_` обязательно. Стартовый триггер свежести (что делать, если устарело) — в CLAUDE.md (always-on); здесь не дублируется.
5. **Связь с wiki — markdown-ссылками.** Пункт, ссылающийся на знание, делает это через `[Название](wiki/type/slug.md)`. Знание живёт в wiki; в STATE — формулировка и ссылка.
6. **Буфер «завершено».** Висит неделю, потом схлопывается: породившее wiki-страницу → ссылка на неё; под git рутина без следа → строка «N задач за период …, см. git log»; без git — удаляется. Память о работе — в wiki/git, не в STATE.
7. **STATE не canonical.** При конфликте с wiki/кодом — правда там. Lint может пометить расхождение, не правит автоматически.
