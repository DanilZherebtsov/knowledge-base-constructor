# Роли — специализированные чаты-сотрудники

Роль — профиль чата для одной функции (примеры — <<СЛОТ ROLES-FILL: роли класса, напр.: юрист, финансист, методолог, security-reviewer>>): «нанять» функцию на задачу, не размывая соседними. Определения — в `roles/<роль>.md`; образец — [roles/_шаблон.md](../roles/_шаблон.md).

## Создание роли — «создай роль <название>»

**Один гибкий процесс, не разные режимы.** Что бы человек ни дал на вход — только фразу, файл-описание роли или ещё и пачку опорных документов — результат один: появляется `roles/<роль>.md`, существующая вики размечается срезом роли, переданные материалы становятся знанием. Меняется только полнота входа, не механика. Нетривиальные мутации — с подтверждением человека.

1. **Собрать описание роли** из данного: только промпт → синтезировать (добрать 2–4 вопросами, если куцо); файл-описание → положить в `raw/`, синтезировать из него; и то и другое → скомбинировать. **Вопросы человеку — только бизнес-уровень** (функция, границы, какие материалы есть); **срез (префиксы/теги) и куда писать находки Claude выводит сам** по домену и [page-conventions.md](page-conventions.md) — `wiki/` внутренний инструмент агента, о тегах человека не спрашивают; роль пишет в уместный для находки тип, не в один. Записать в `roles/<роль>.md` по [roles/_шаблон.md](../roles/_шаблон.md) (поля заполняет Claude): зона ответственности, границы, **срез (префиксы/теги)**, что читать в начале, куда писать находки.
2. **Гигиена тегов/префиксов.** Перед фиксацией среза — сверка с существующими (`grep -rho 'tags:' wiki/`, `ls wiki/<type>/`) и **переиспользование**, а не плодёж синонимов. Новый тег/префикс — только если подходящего нет.
3. **Засев знания — если даны опорные документы** (стандарт, регламент, нормативы функции). Они не загружены заранее: человек указывает на них, Claude кладёт в `raw/`, детально изучает и обычным [ingest](ingest.md) синтезирует в профильный type под префиксом/тегом роли.
4. **Init — всегда, на любом входе.** Роль часто заводят в середине проекта, когда работа по домену уже осела в вике без её тега. Claude один раз проходит по всей вике, находит релевантные роли страницы, показывает человеку список-предложение и на подтверждённых проставляет тег (при необходимости переименовывает под префикс), освежает `updated:`, дописывает строку в `wiki/log.md`. **Завершить коротким синтезом для человека:** что отнесено к роли и под каким тегом/префиксом (напр. «отнёс к роли 3 страницы: …; проставил доменный тег»); ключевые найденные страницы записать в файл роли в «что читать в начале». Без init поздно заведённая роль не увидит прошлую работу по домену.
5. **Объявить срез** и что роль вызывается фразой «работаем в роли <название>».

**CLAUDE.md менять не нужно:** механизм ролей в нём описан, новая роль подхватывается по своему файлу, список ролей — `ls roles/`.

## Вызов

«работаем в роли <роль>» (путь называть не нужно — Claude сам открывает `roles/<роль>.md`), дальше — задача. Claude читает файл роли, `CLAUDE.md` и стартовые страницы вики из роли, затем работает строго в зоне роли (выходит — только чтобы перенаправить).

## База знаний роли — срез общей вики, а не отдельная вика

У роли нет своего хранилища. Её «база знаний» — **срез одной общей вики** (по префиксам/тегам) плюс связанные сырые файлы. Почему не отдельная вика: решения опираются на знание из разных функций сразу — оно должно лежать в одном графе; отдельная вика на функцию рвёт связи и плодит дубли.

## Ретрив и сохранение

- **Ретрив (не вчитывая всю вику):** срез задан префиксами/тегами роли. Claude находит страницы по `wiki/index.md` и префиксу имени (`ls <type>/<префикс>-*`), при необходимости — `grep` по `tags:`; читает только совпавшие, на шаг по ссылкам. Новые непомеченные доменные страницы подхватываются повторным init-проходом по запросу.
- **Сохранение:** новые находки — в общую вику обычным [ingest](ingest.md) (с `sources:`, человек подтверждает); черновой разбор — в рабочий слой класса; решение — ADR в `decisions/`.
- **Дообучение на любом этапе:** засев (шаг 3 создания) не разовый — меняется стандарт/практика функции, человек кладёт новые документы и просит роль их обработать; роль инджестит под своим тегом, замещённые версии помечаются `superseded` (история не теряется). Роль дообучается на ходу, без переписывания файла роли.

## Правила

- Роль не пишет в вики молча: через [ingest](ingest.md) с источником.
- Роль не решает за владельца: готовит рекомендацию; выбор и ADR — через lifecycle-файл класса.
- Роли заводит и правит человек (образец `_шаблон.md`); это рабочая область, не методология.
