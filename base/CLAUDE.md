Базовая идея: не выводить одно и то же знание из сырых материалов заново при каждом вопросе. Вместо этого — однократно компилировать каждый источник в постоянную, взаимосвязанную вики. Дальше Claude читает вики, а не сырые материалы, когда отвечает. К сырому возвращается только тогда, когда нужно интегрировать новые данные или разрешить противоречие.

Корневой файл содержит **always-on правила и указатели**. Детальные процедуры — в [methodology/](methodology/), читаются по триггеру.

> **Это base-template — скелет.** Производный шаблон класса заполняет все `<<СЛОТ: …>>` (список — в [README.md](README.md)) и добавляет доменный lifecycle-файл. Остальное наследуется без изменений.

## Архитектура

```
input/           ← Свалка для входящих. Кидай сюда любые новые материалы как есть —
                   на «обработай» Claude разложит их по raw/ (тип + имя) и усвоит в wiki/.
                   После обработки пустеет. Это не архив (архив — raw/).
raw/             ← Сырые источники, read-only. Подпапки заводит bootstrap под источники
                   проекта и вписывает сюда; растут по мере надобности (глубина свободна).
wiki/            ← Скомпилированные знания. Управляет Claude. Плоско, глубина = 1.
  decisions/     (ADR — что решили и почему; связаны через supersession)
  discovery/     (знание о внешнем мире проекта; группировка префиксами имени)
  synthesis/     (записанные обратно ответы, сквозные анализы)
  principles/    (правила из инцидентов; читаются перед нетривиальной задачей)
                 <<СЛОТ S2: центральный/доменный тип(ы) класса — напр. architecture/, claims/, entities/;
                  при необходимости убрать неподходящие из дефолтных выше>>
  index.md       (каталог — одна строка на страницу)
  log.md         (лог операций, только дополнение)
methodology/     ← Инструкции для Claude (по триггеру). Часть шаблона.
  ingest.md  query.md  lint.md  page-conventions.md
  state-rules.md  index-log-format.md  bootstrap.md  roles.md
                 <<СЛОТ S6: доменный lifecycle-файл — spec-/question-/decision-lifecycle.md>>
roles/           ← Определения ролей-чатов. `_шаблон.md` — образец.
output/          ← Корень рабочих файлов — **есть у всех классов**. Создаётся пустым;
                   подпапки (drafts/, черновики/, папки под живые документы) — по факту.
                 <<СЛОТ S4: дополнительные рабочие слои для классов с кодом — specs/ + src/ + data/>>
CLAUDE.md        ← Этот файл.
STATE.md         ← Операционное состояние (намерения, не факты; не canonical).
```

Правила:

- **`raw/` неизменяем.** Только добавление. Оригинальная формулировка важна при перепроверке.
- **`wiki/` под управлением Claude.** Человек читает, но не правит руками. Растёт через: (а) ingest источника из `raw/`; (б) запись-обратно ответа в `synthesis/` после query; (в) извлечение ADR из принятого решения; (г) запись принципа из инцидента. Если в вики ошибка — исправьте источник в `raw/` (или укажите Claude), и он перекомпилирует.
- **Глубина wiki = 1.** Один уровень тематических подпапок (типы выше), глубже не идти. 30+ однородных страниц в одном типе → горизонтальное расширение (новый top-level type или префиксы имён), не подпапки.
- **`raw/` — исключение.** Внутри `raw/` глубина по необходимости разрешена (хранилище под навигацию человеком).
- **`methodology/` — часть шаблона, не рабочая область.** Правки только человеком при пересмотре методологии.
- <<СЛОТ S7: правило авторитета — «**Код побеждает вики**» (классы с `src/`) ИЛИ «**Источники побеждают вики**» (классы без кода; для research добавить «**каждая цитата с локализацией**, lint-проверяемо»)>>
- **git опционален.** Под версионным контролем — Claude коммитит после ingest/bootstrap; иначе история в `wiki/log.md` и датах страниц, шаги «закоммить» пропускаются.
- **Структура растёт по мере надобности.** Не подошёл источник под существующие подпапки `raw/` — заведи новую (в `raw/` глубина свободна; в `wiki/` — плоско). Изменил раскладку — отрази в дереве этого файла и в затронутых контекстных файлах.
- **Имена файлов.** Описательные, через подчёркивание/kebab-case. Сырые файлы префиксуются `YYYY-MM-DD-`, если известна дата источника.

## Иерархия источников

1. **CLAUDE.md** — always-on правила и контекст.
2. **`methodology/`** — детали операций. По триггеру, не в фоне.
3. **Вики** (`wiki/`) — canonical-знание из ingest.
4. **Сырые источники** (`raw/`) — первоисточник при сомнениях.
5. **Auto-memory** (`MEMORY.md`, `memory/`) — кэш между сессиями, **не canonical**.

**memory vs вики:** всегда верить вики. **вики vs источник:** перепроверить источник и исправить вики (не предполагать, что вики «знает лучше» — так заводится дрейф). **Запись в memory:** новое знание сначала в `wiki/` через ingest; в memory — только короткий указатель.

**`STATE.md` — отдельная ось**, не часть иерархии: намерения и прогресс, а не утверждения о мире. С wiki не конфликтует (разные домены).

## Операционное состояние

`STATE.md` в корне — единственное место для текущих планов и прогресса. **Структура — фиксированный набор разделов** (<<СЛОТ S5: список разделов класса; механика — в [methodology/state-rules.md](methodology/state-rules.md)>>). Пустые разделы остаются с пометкой `_пусто_`.

**Триггеры (для Claude):**
- При старте сессии — молча читать STATE.md (контекст «на чём остановились»).
- На вопросы «на чём остановились / что в работе / что дальше / блокеры» — STATE.md основной источник.
- Если `_Обновлено:_` старше 7 дней — в первой реплике предложить «STATE устарел на N дней, что изменилось?».

**Все стартовые проверки молчаливы.** Свежесть STATE, свежесть lint (§5 «Дисциплины»)<<СЛОТ DEADLINE-CHECK: классы с календарём обязательств (механика decision-lifecycle) добавляют « и ближайшие/просроченные обязательства календаря»; иначе пусто>> упоминаются в первой реплике **только при отклонении**. ## Роли (специализированные чаты-сотрудники)

В `roles/` лежат описания ролей — профили чатов для одной функции. «работаем в роли <роль>» → Claude открывает `roles/<роль>.md` и работает строго в зоне роли, пишет находки в свой срез общей вики (по префиксам/тегам) обычным ingest. Роль — линза над **одной** общей викой, не отдельная вика. Завести — «создай роль <название>» (гибкий процесс: срез + разметка существующей вики под роль). Механика — [methodology/roles.md](methodology/roles.md).

## Дисциплина (то, что не даёт вики сгнить)

1. **Фильтровать на входе.** В вики попадает только то, что готов защищать.
2. **Supersession вместо тихого исчезновения.** Старое остаётся, помеченное `superseded` и связанное с заменой.
3. **Никакой ложной точности.** Без числовых оценок уверенности — достоверность видна из цепочки источников.
4. **Человек в петле записи.** Claude предлагает; человек подтверждает нетривиальную мутацию вики.
5. **Профилактика (lint) не опциональна.** Раз в неделю. Claude при старте читает дату последней `lint` в `wiki/log.md`; > 7 дней — предлагает прогон.
6. **Сначала схема, потом механизм.** Что-то не так — сначала правь этот файл или `methodology/`, не плоди обходные пути.
7. **Расширение схемы — только по горизонтали.** Новый top-level type в `wiki/` (плоско) или новая top-level папка. Углублять типы запрещено. `raw/` — исключение.
8. **Синтез знаний при закрытии единицы работы.** Когда единица работы закрывается, Claude обязательно ревизует, какое новое знание она породила, и предлагает занести его в `wiki/` по всем релевантным типам — не только профильному. Человек подтверждает (ADR/принципы — никогда молча). Пропуск = вики отстаёт от того, что мы реально знаем. Что такое «единица работы» — задаёт доменный lifecycle (<<СЛОТ S6>>).

## Принципы работы Claude в задачах

Каждая нетривиальная задача проходит две фазы: сначала **остановись и подумай**, потом действуй.

**Нетривиальная** — где нужен выбор (между подходами, формулировками) или план (несколько шагов/файлов). **Тривиальная** (фаза «подумай» не нужна): опечатка, переименование, пересказ одной страницы. В сомнении — считать нетривиальной.

### Перед началом — фаза «подумай»

**Закрой все запросы в сообщении (даже для тривиальных задач).** Если просьб несколько — перечисли их списком в начале ответа и закрой каждую; конкретные action-items (команды, пути) давай первыми, до анализа. Не зарывайся в один под-вопрос, теряя остальные — повторяющаяся ошибка.

1. **Перечитай релевантные принципы** (`wiki/principles/<applicability>.md`) — извлечённые из прошлых случаев правила.
2. **Сформулируй предположения.** Неопределённость — спрашивай, не угадывай.
3. **Покажи разные трактовки**, если запрос двусмыслен.
4. **Не выдумывай.** Цифры и факты — только из источников/вики; нет данных — `[требует уточнения]`.
5. **Один вариант за раз**, начиная с простого.
6. **Возражай, когда оправдано**, а не выполняй молча плохое решение.

### В процессе — фаза «действуй»

1. **Одна задача за раз.** 2. **Сначала простота.** 3. **Точечные изменения.** 4. **Выполнение от цели** (критерии успеха до начала). 5. **На языке человека** — объясняй действием и пользой, не внутренней кухней (имена инструментов, песочница/доступы, технические причины, названия папок/типов/операций как термины). Нужно разрешение или что-то не вышло — скажи смысл простыми словами.

### После — фиксация принципа

Родилось правило «всегда X / никогда Y» — предложить записать в `wiki/principles/<applicability>.md` с источником. Только из конкретных случаев, не из общих соображений.
